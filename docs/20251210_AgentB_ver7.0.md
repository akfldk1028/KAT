## 1. 개요 및 역할

### 1.1 Agent B란?

**정의**: Kanana 온프레미스 LLM 기반 실시간 피싱 탐지 및 사용자 보호 에이전트

**핵심 가치**:
- **안심 가드**: 사용자를 피싱 사기로부터 보호하는 신뢰할 수 있는 가드
- **실시간 탐지**: 메시지 수신 즉시 위험도 분석 및 경고
- **투명한 설명**: 왜 위험한지 명확한 근거와 함께 제공
- **실제 사례 기반**: 금감원/KISA/경찰청 통계 기반 9개 유형 탐지

**차별점 비교**:

```
┌─────────────────────────────────────────────────────────────┐
│ 기존 시스템 (Rule-based)                                       │
├─────────────────────────────────────────────────────────────┤
│ Rule(주인공) → AI(보조)                                        │
│ ├─ 고정 4-Stage 파이프라인                                     │
│ ├─ 모든 메시지 동일 처리                                       │
│ └─ AI는 System Prompt만 제공                                  │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ Hybrid Agent (AI-driven) - Agent B ver7.0                   │
├─────────────────────────────────────────────────────────────┤
│ Kanana LLM(주인공) → Rule(도구)                               │
│ ├─ ReAct Loop: Thought → Action → Observation               │
│ ├─ 상황별 선택적 도구 실행                                     │
│ ├─ 9개 실제 스미싱 유형 학습                                   │
│ └─ Kanana LLM이 맥락 기반 최종 판단                            │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 핵심 역할

**Who (대상)**:
- 카카오톡 사용자 5천만명
- 일간 1천만 메시지 교환
- 연간 피싱 피해 3천억원 (카카오톡 비중 30%)

**What (기능)**:
1. **실시간 위험도 분석**: CRITICAL / DANGEROUS / SUSPICIOUS / SAFE
2. **9개 유형 분류**: A-1~C-3 (실제 수집 스미싱 유형)
3. **맥락 기반 판단**: DB 신고 이력 + 대화 관계 + 패턴 분석 종합
4. **투명한 설명**: 추론 과정 전체 공개 (XAI)
5. **행동 권장**: "직접 전화 확인", "절대 송금 금지" 등

**Why (필요성)**:
- 기존 Rule 기반 시스템: 오탐률(False Positive) 12%, 미탐률(False Negative) 18%
- 새로운 피싱 수법 매주 등장 → Rule 업데이트 지연
- 맥락 무시 → 1개월 대화 이력도 DB 신고 하나에 차단

### 1.3 ver7.0 Hybrid Agent 설계 철학

**핵심 원칙**:

**ReAct Loop란?**
- **Re**asoning(사고) → **Act**ion(행동) → Observation(관찰)을 반복하는 AI 판단 패턴
- Kanana LLM이 상황을 분석하고 → 필요한 도구를 선택하여 → 결과를 관찰한 후 다음 판단
- 최대 5 사이클 반복하여 충분한 증거 수집

---

1. **실제 사례 우선 (Real Case First)**
   - Cialdini 추상 원리 → 금감원/KISA 통계 기반 9개 유형
   - 각 유형별 정부 공식 통계 명시
   - 예: A-1 (가족사칭) → 금감원 2023: 33.7%

2. **맥락 우선 (Context First)**
   - 1개월 대화 이력 > DB 신고 1건
   - 평소 톤 일관성 > 패턴 매칭
   - 예: DB 신고 + 1개월 가족 대화 → "번호 재활용" 추론

3. **비판적 사고 (Critical Thinking)**
   - DB 신고 = 사전 확률(Prior Probability), 절대 진리 아님
   - 대화 이력 = 증거로 사후 확률(Posterior Probability) 재계산

4. **설명 가능성 (Explainability)**
   - 모든 판단에 reasoning 제공
   - decision_process로 추론 과정 투명화

5. **보수적 판단 (Conservative)**
   - 신뢰도(confidence) < 0.7 → SUSPICIOUS (사용자 확인 요청)
   - 불확실성 높음 → 차단보다 경고

## 2. 작동 구조 (Architecture)

### 2.0 전체 작동 로직 (구조화 총정리)

#### 2.0.1 작동 구조 Overview

**전체 흐름 다이어그램**:

```
┌─────────────────────────────────────────────────────────────────┐
│                    Agent B ver7.0 전체 작동 흐름                   │
└─────────────────────────────────────────────────────────────────┘

[INPUT] 카카오톡 메시지 + 대화 이력
   ↓
┌──────────────────────────────────────────┐
│  Phase 1: ReAct Loop (최대 5 사이클)      │
│  ▸ Kanana LLM이 상황 판단 → 도구 선택    │
│  ▸ 9개 스미싱 유형 학습                  │
└──────────────────────────────────────────┘
   ↓
┌─────────────────────────────────────────────────────────────────┐
│  Cycle 1: 🤔 Thought → Pattern Analysis (9개 유형)             │
│  ├─ Kanana LLM: "가족사칭 패턴 의심 (A-1)"                      │
│  └─ Action: pattern_detection_mcp 호출                         │
│      └─ Observation: {category: "A-1", confidence: 0.92}       │
│         matched: ["엄마", "급해", "폰 고장"]                     │
│         source: "금감원 2023: 가족사칭 33.7%"                   │
└─────────────────────────────────────────────────────────────────┘
   ↓
┌─────────────────────────────────────────────────────────────────┐
│  Cycle 2: 🤔 Thought → Entity Extraction                       │
│  ├─ Kanana LLM: "계좌번호 추출 필요"                            │
│  └─ Action: entity_extractor_mcp 호출                          │
│      └─ Observation: {accounts: ["110-123-456789"]}            │
└─────────────────────────────────────────────────────────────────┘
   ↓
┌─────────────────────────────────────────────────────────────────┐
│  Cycle 3: 🤔 Thought → DB Verification                         │
│  ├─ Kanana LLM: "블랙리스트 조회"                               │
│  └─ Action: threat_intelligence_mcp 호출                       │
│      └─ Observation: {db_prior: 0.15, found: false}            │
└─────────────────────────────────────────────────────────────────┘
   ↓
┌─────────────────────────────────────────────────────────────────┐
│  Cycle 4: 🤔 Thought → Trust Analysis                          │
│  ├─ Kanana LLM: "대화 이력 확인"                                │
│  └─ Action: trust_indicator_mcp 호출                           │
│      └─ Observation: {trust_score: 0.25, days: 0}              │
└─────────────────────────────────────────────────────────────────┘
   ↓
┌──────────────────────────────────────────────────────────────────┐
│  Phase 2: 동적 추론 기반 최종 판단                                │
│  ▸ Kanana LLM이 맥락을 이해하고 지능적으로 판단                   │
└──────────────────────────────────────────────────────────────────┘
   ↓
┌─────────────────────────────────────────────────────────────────┐
│  Step 1: 증거 수집 & 품질 평가                                   │
│  ├─ pattern_conf = 0.92 (A-1: 가족사칭)                        │
│  ├─ db_prior = 0.15 (DB 신고 없음)                             │
│  ├─ trust_score = 0.25 (대화 이력 없음)                        │
│  └─ evidence_quality = {days: 0, db_sources: 0}                │
└─────────────────────────────────────────────────────────────────┘
   ↓
┌─────────────────────────────────────────────────────────────────┐
│  Step 2: 맥락 기반 가중치 동적 조정                              │
│  ├─ Kanana 판단: "패턴만 강력 + DB/Trust 약함"                 │
│  ├─ 가중치 조정: 0.4/0.3/0.3 → 0.6/0.2/0.2                     │
│  └─ 이유: "신종 수법 대응 → 패턴 중심 판단"                      │
└─────────────────────────────────────────────────────────────────┘
   ↓
┌─────────────────────────────────────────────────────────────────┐
│  Step 3: 사후 확률 계산 (동적 가중치 적용)                       │
│  └─ posterior = 0.6×0.92 + 0.2×0.15 + 0.2×(1-0.25) = 0.732     │
└─────────────────────────────────────────────────────────────────┘
   ↓
┌─────────────────────────────────────────────────────────────────┐
│  Step 4: 신뢰 구간 계산 (불확실성 추정)                          │
│  └─ CI = [0.607, 0.857], uncertainty = 0.25                    │
└─────────────────────────────────────────────────────────────────┘
   ↓
┌─────────────────────────────────────────────────────────────────┐
│  Step 5: 추론 기반 위험도(Risk Level) 판정                       │
│  ├─ 기본 판정: posterior 0.732 → SUSPICIOUS                    │
│  ├─ Kanana 추론: "경계선 but 패턴 강력 → 상향"                  │
│  └─ 최종 판정: DANGEROUS (73.2%, 신뢰도 75%)                   │
└─────────────────────────────────────────────────────────────────┘
   ↓
[OUTPUT] 최종 결과
{
  "final_risk": "DANGEROUS",
  "category": "A-1",
  "confidence": 0.75,
  "user_message": "⚠️ 가족 사칭 의심...",
  "government_source": "금감원 2023: 가족사칭 33.7%"
}
```

#### 2.0.2 핵심 구성 요소 (3-Layer Architecture)

**아키텍처 계층 구조**:

```
┌──────────────────────────────────────────────────────────────┐
│  Layer 1: ReAct Loop Engine (Kanana LLM 자율 판단)           │

│  • 5개 MCP Tools (Action 실행)                               │

│  • 최대 5 사이클 반복                                         │

         ↓ observations (증거 리스트)
┌──────────────────────────────────────────────────────────────┐
│  Layer 2: 동적 추론 엔진 (Kanana 판단 로직)                   │

│  • 불확실성 추정 (신뢰 구간 계산)                             │
│  • 증거 정렬 분석 (충돌 검출)                                 │
│  • 추론 기반 위험도 판정                                      │
└──────────────────────────────────────────────────────────────┘
         ↓ risk_level + reasoning
┌──────────────────────────────────────────────────────────────┐
│  Layer 3: XAI 설명 생성 (투명성 보장)                         │
├──────────────────────────────────────────────────────────────┤
│  • 가중치 조정 이유 설명                                      │
│  • 사후 확률 + 신뢰 구간 표시                                 │
│  • 위험도 판단 근거 제시 (9개 유형 + 정부 통계)               │
│  • 최종 신뢰도 계산                                           │
└──────────────────────────────────────────────────────────────┘
```

#### 2.0.3 5개 MCP Tools 역할 분담

| Tool | 역할 | 출력 | 호출 타이밍 |
|------|------|------|------------|
| **pattern_detection** | **9개 유형 분류** | category (A-1~C-3, NORMAL), confidence, **정부 통계** | Cycle 1 (필수) |
| **entity_extractor** | 계좌/URL 추출 | accounts, urls, phones | Cycle 2 (선택적) |
| **threat_intelligence** | DB 조회 | db_prior, blacklist 여부 | Cycle 2-3 (선택적) |

**특징**:
- ✅ Kanana LLM이 **맥락에 따라 선택적 호출** (고정 순서 아님)
- ✅ 정상 메시지는 **1-2 사이클**로 조기 종료
- ✅ 의심 메시지는 **3-5 사이클**로 충분한 증거 수집
- ✅ **9개 유형별 정부 통계** 제공 (금감원/KISA/경찰청)

## 3. 스미싱 9개 유형 분류

### 3.1 분류 철학

**금감원 공식 철학**: 사칭형 (Impersonation) vs 사기형 (Fraud) + 실용적 세분화

**출처**:
- 금융감독원 (2023). 보이스피싱 피해 통계
- 한국인터넷진흥원 (2024). 2분기 스미싱 공격 분석
- 경찰청 (2022-2023). 전기통신금융사기 통계

---

### 3.2 대분류 1: 사칭형 (Impersonation)

#### A그룹: 개인 관계 사칭

**심리 기제**: 호감(Liking) + 긴급성(Urgency) + 동정심

| 코드 | 카테고리 | 대표 패턴 | 키워드 예시 | Cialdini | 정부 통계 |
|------|---------|----------|------------|----------|----------|
| **A-1** | 지인 및 가족 사칭 | 액정 파손, 급전 요청 | "엄마", "아빠", "폰 고장", "급해", "계좌" | Liking + Urgency | 금감원 2023: **33.7%** (2위) |
| **A-2** | 경조사 빙자 | 청첩장, 부고장 | "청첩장", "부고", "결혼식", "조의금" | Social Proof | KISA 2023: **1.7만건** (4,250배↑) |
| **A-3** | 로맨스 스캠 | 이성 교제 → 금전 요구 | "만나고 싶어", "선물", "도와줘" | Liking + Reciprocity | 경찰청 2023: 신종 수법 |

**정부 공식 근거**:
- 금감원/경찰청 합동 보도자료: "메신저 피싱 주의" 소비자경보 (2018-2호)
- 2023년 피해 비중: 가족사칭 33.7% (전체 2위)
- KISA: 부고장/청첩장 스미싱 2023년 1.7만건 (전년 대비 4,250배 ↑)

---

```python
def calculate_trust_indicator(sender_id: str, history: list) -> dict:
    """
    신뢰 지표 계산

    근거: 금감원 2023 통계 기반 가중치 설계
    - 초면 사칭: 92% → 대화 기간이 가장 중요한 변수
    - 장기 관계 (>30일): FP 위험 90% 감소
    - 연락처 저장: 보조 지표
    """

    if not history:
        return {
            "trust_score": 0.08,  # 초면 사칭 92% → (1-0.92) = 0.08
            "conversation_days": 0,
            "message_count": 0,
            "is_new_contact": True,
            "is_contact_saved": check_contact_saved(sender_id),
            "source": "금감원 2023: 초면 사칭 92%"
        }

    # 대화 기간 계산
    first_msg = min(msg["timestamp"] for msg in history)
    last_msg = max(msg["timestamp"] for msg in history)
    days = (last_msg - first_msg) / 86400

    # 메시지 개수
    message_count = len(history)

    # 연락처 저장 여부
    is_saved = check_contact_saved(sender_id)

    # 정규화 기준 (금감원 2023 통계 기반)
    # - 30일 기준: "장기 관계 (>30일) FP 90% 감소" 통계
    # - 50개 메시지: 경험적 (월 1-2회 대화 × 30일)
    factors = {
        "days_factor": min(days / 30, 1.0),         # 30일 기준
        "msg_factor": min(message_count / 50, 1.0),  # 50개 기준
        "saved_factor": 1.0 if is_saved else 0.0
    }

    # 가중 평균 (금감원 2023 통계 반영)
    # 초면 vs 장기 차이 92% → 대화 기간 가중치 0.7
    trust_score = (
        0.7 * factors["days_factor"] +    # 금감원 2023: 초면 사칭 92%
        0.2 * factors["msg_factor"] +     # 보조 지표
        0.1 * factors["saved_factor"]     # 보조 지표
    )

    return {
        "trust_score": trust_score,
        "conversation_days": days,
        "message_count": message_count,
        "is_new_contact": days < 1,
        "is_contact_saved": is_saved,
        "source": "금감원 2023: 초면 사칭 92%, 장기(>30일) FP 90%↓"
    }
```
